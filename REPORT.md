**Виконав:** Яремчук Павло, студент групи ШІ-32

**Проєкт:** Аналітика звільнення працівників (Employee Churn Prediction)

**Версія проєкту:** v0.1.0

Метою даної роботи було створення повноцінного проєкту з аналізу даних та машинного навчання з використанням системи контролю версій Git. У процесі роботи було імітовано реальний процес командної розробки: від базового налаштування структури репозиторію до паралельної роботи в гілках, вирішення конфліктів та випуску релізу. 

## Хід виконання роботи

### 1. Ініціалізація репозиторію та налаштування середовища
* Створено віддалений репозиторій `open-data-ai-analytics` на GitHub та зв'язано його з локальним середовищем.
* Налаштовано ізольоване віртуальне середовище `.venv` (Python 3.12) для уникнення конфліктів бібліотек (pandas, scikit-learn, seaborn, matplotlib).

### 2. Побудова структури проєкту
Створено стандартизовану ієрархію папок та файлів для Data Science проєктів:
* `data/raw/`
* `notebooks/`
* `src/`
* `reports/figures/`

### 3. Налаштування ігнорування файлів (.gitignore)
Створено файл `.gitignore`, куди було додано системні директорії (`__pycache__/`, `.idea/`), файли середовища (`.venv/`) та папку із сирими даними (`data/raw/`), щоб запобігти потраплянню великих та системних файлів на сервер GitHub.

### 4. Оформлення документації (README.md)
У головний `README.md` було додано:
* Назву та мету проєкту.
* Посилання на джерело відкритих даних.
* Три ключові гіпотези для подальшого аналізу.

### 5. Базовий функціонал (Гілка feature/data_load)
Для розробки функціоналу завантаження даних було створено окрему гілку `feature/data_load`. 
* Написано скрипт `src/data_load.py`, який читає `.csv` файл за допомогою бібліотеки `pandas` та виводить базову інформацію про успішне завантаження.
* Після тестування гілку було успішно злито (merged) у головну гілку `main`.

### 6. Паралельна розробка та Pull Requests
Проведено імітацію одночасної роботи кількох розробників. Було створено дві незалежні гілки від `main`:
1. **`feature/data_quality_analysis`**: Створено скрипт `data_quality.py` для перевірки датасету на пропуски (NaN) та дублікати.
2. **`feature/data_research`**: Створено скрипт `data_research.py`. Виконано аналітичну перевірку трьох гіпотез шляхом групування даних. Крім того, побудовано модель машинного навчання (Random Forest Classifier), яка прогнозує звільнення працівника з оцінкою точності (Accuracy).
Обидві гілки були відправлені на GitHub (push), де для них було створено Pull Requests з детальним описом змін. Після Code Review гілки були успішно злиті в `main`.

### 7. Вирішення Merge Conflict
Для закріплення навичок роботи з Git було штучно ініційовано конфлікт злиття (merge conflict). 
* Дві нові тестові гілки паралельно змінили перший рядок (заголовок) у файлі `README.md`.
* Під час спроби злиття обох гілок у `main` Git видав попередження про конфлікт.
* Конфлікт було вирішено вручну в редакторі коду шляхом видалення маркерів `<<<<<<<`, `=======`, `>>>>>>>` та збереження фінального коректного варіанту тексту. Зміни зафіксовано новим комітом.

### 8. Візуалізація даних (feature/visualization)
В окремій гілці `feature/visualization` додано скрипт `src/visualization.py`. З використанням бібліотеки `seaborn` було згенеровано три графіки, що візуально підтверджують висунуті гіпотези. Графіки автоматично зберігаються у папку `reports/figures/` у форматі `.png`. Гілку злито в `main`.

### 9. Фінальний реліз та тегування
Для фіксації першої стабільної версії проєкту:
* Створено файл `CHANGELOG.md` з описом усіх доданих функцій.
* Створено Git-тег `v0.1.0`.
* Усі фінальні зміни та теги були відправлені на віддалений репозиторій.

## Висновок
У результаті виконання завдання було створено повноцінний ML-пайплайн. Отримано практичний досвід роботи з системою контролю версій Git: створення та перемикання гілок, локальне та віддалене злиття (через PR), розв'язання конфліктів, налаштування середовища розробки та структурування Python-проєктів.
